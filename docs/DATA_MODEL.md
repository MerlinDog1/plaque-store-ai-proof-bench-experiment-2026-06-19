# Data Model

This prototype is still a proof-first frontend with local/mock persistence. The
practical production target is Supabase Postgres plus Supabase Storage, with
Stripe handling payment, Resend handling email, and Gemini handling AI-assisted
proof composition. Do not store real service credentials in the repository.

## Core Principles

- Customers do not need an account or password to design, approve, or pay.
- Every shared proof is addressable by an opaque public token, not by a numeric
  database id.
- The live proof state is saved as structured JSON so the app can reopen a
  design exactly as the customer saw it.
- Generated SVG/PDF/raster files live in storage buckets; rows store metadata
  and storage paths.
- Payment state comes from Stripe webhooks, not from client-side success pages.
- Quotes are first-class flags on carts/orders, not a separate manual workflow.

## Order Statuses

Recommended `orders.status` values:

- `draft`: created before checkout/payment is confirmed.
- `awaiting_payment`: Stripe checkout started.
- `payment_failed`: Stripe reported failure or expiration.
- `paid`: payment confirmed by webhook, production not yet started.
- `proof_review`: paid but needs manual review before production.
- `production_ready`: proof/artifacts passed checks and can be made.
- `in_production`: sent to supplier or internal production.
- `quality_check`: produced item is being checked.
- `dispatched`: shipped to customer.
- `complete`: delivered or administratively closed.
- `cancelled`: cancelled before fulfilment.
- `refunded`: fully refunded.

Keep `payment_status` and `fulfilment_status` separate so payment refunds do not
overwrite production history.

## Quote Flags

Use a JSON object for transparent rule output, for example:

```json
{
  "requires_manual_quote": false,
  "reasons": [],
  "oversized_bed": false,
  "non_uk_delivery": false,
  "custom_material": false,
  "custom_fixing": false,
  "complex_artwork": false,
  "quantity_discount": false,
  "supplier_review": false
}
```

Initial automatic flags:

- `oversized_bed`: plaque does not fit the standard production bed.
- `non_uk_delivery`: delivery address is outside the UK.
- `custom_material`: material is not in the current fixed price list.
- `custom_fixing`: fixing/backing combination is not production-standard.
- `complex_artwork`: future photo/vector/artwork flow is present.
- `quantity_discount`: quantity is high enough for manual pricing.
- `supplier_review`: anything the pricing function cannot confidently cost.

If `requires_manual_quote` is true, the customer can still save/email the proof,
but Stripe checkout should be replaced with a quote request flow.

## Storage Buckets

Recommended Supabase Storage buckets:

- `proof-artifacts`: customer proof SVGs, preview PNGs, realistic preview PNGs.
- `production-artifacts`: final production SVG/PDF files, private.
- `customer-uploads`: future uploaded logos/photos/reference files, private.
- `material-assets`: approved app texture scans, public read or CDN-backed.
- `email-assets`: small generated thumbnails used in Resend templates.

Suggested path pattern:

```text
proof-artifacts/{proof_session_id}/{artifact_type}-{timestamp}.{ext}
production-artifacts/{order_id}/{order_item_id}/production.svg
customer-uploads/{customer_id_or_token}/{upload_id}/{filename}
```

## Row-Level Security Shape

For the no-login MVP, avoid broad authenticated-user policies. Use server-side
routes for writes and signed/magic tokens for reads.

- Public client may create/update anonymous proof sessions through an API route.
- Public client may read a proof only by `public_token`.
- Production artifacts are never public; use short-lived signed URLs for admins.
- Stripe and Resend webhooks use service-role access only from server routes.
- Future admin UI should use Supabase Auth with an `admin_users` allowlist.
